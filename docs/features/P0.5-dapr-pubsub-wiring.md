# P0.5 — Wire Dapr pubsub end-to-end

## Problem

`deploy/k8s/base/infra/dapr/pubsub.yaml` is declared but unused. `pkg/outbox` is wired into `order` and publishes `order.created` to a relay sink, but no consumer reads from the broker. Every downstream feature in Phases 0-3 (stock reservation, saga, notifications, search indexer, audit log, reporting) needs working pubsub. This is the single biggest infrastructure unlock — until it lands, the rest of P0 is dead code.

## User stories

- As a **platform engineer**, I want a producer-side `pkg/outbox` write in service A to deterministically reach a consumer-side `pkg/outbox` inbox handler in service B, so that domain events propagate without bespoke broker code in biz layers.
- As an **operator**, I want broker outages to backpressure the outbox relay (not the user-facing RPC) and for messages to redeliver after broker recovery, so that no events are lost.
- As a **service author**, I want one consistent way to consume events — a single `Subscribe(topic, handler)` call wired through Wire — so that I don't have to learn the Dapr SDK to add a consumer.

## Acceptance criteria

- [ ] A broker is selected and provisioned via Helm in `deploy/helm/`. Decision: **Redis Streams** (existing Redis already in chart; lowest ops cost; sufficient for current event volume). Revisit at >5K events/sec.
- [ ] `pubsub.yaml` Dapr component points at the chart-provisioned Redis with auth from the existing `secretstore`.
- [ ] `pkg/outbox/relay.go` publishes via Dapr (`POST /v1.0/publish/<pubsub>/<topic>`) using a CloudEvents 1.0 envelope. Redelivery on transient broker errors is bounded with exponential backoff and a poison queue after N retries.
- [ ] A reusable `pkg/eventbus` (or extension to `pkg/outbox`) exposes `Subscribe(topic, handler)` that registers a Dapr HTTP subscription and runs each delivery through `pkg/outbox.Inbox` for dedup.
- [ ] Each of the 4 services has a stub consumer subscribed to a no-op `health.ping` topic that increments a metric — proof the wiring works, removable after one real consumer ships.
- [ ] A Tilt resource `pubsub-smoke` runs an end-to-end test: producer publishes, consumer receives, idempotent on retry.
- [ ] Broker outage (`kubectl delete pod redis-0`) does not fail user RPCs; outbox relay backs off and resumes after Redis returns.
- [ ] Outbox relay lag and inbox dedup hit rate are exposed as Prometheus metrics.

## Out of scope

- Replacing Dapr with native Kafka/NATS clients — explicit no for this feature.
- Multi-region pubsub or cross-cluster fan-out.
- Schema registry / Avro / protobuf event payloads — JSON is fine for now; payload schemas live alongside topic constants in each service's `internal/biz/events.go`.
- Exactly-once delivery — we ship at-least-once + idempotent consumers (inbox dedup). "Exactly-once" is application-level, not transport-level.

## Implementation notes

### Files / packages
- `pkg/outbox/relay.go` — switch sink to Dapr HTTP publish.
- `pkg/eventbus/subscriber.go` (new) — wraps Dapr subscription registration + inbox dedup.
- `pkg/eventbus/cloudevent.go` (new) — envelope marshal/unmarshal.
- `app/<svc>/internal/server/event.go` (new per service) — kratos HTTP route for Dapr-delivered events.
- `app/<svc>/cmd/server/wire.go` updates.
- `deploy/helm/values.yaml` — Redis password from secretstore.
- `deploy/k8s/base/infra/dapr/pubsub.yaml` — pin connection details.
- `Tiltfile` — add `pubsub-smoke` resource.

### Pattern reference
- `pkg/outbox` already exists with `relay.go`, `inbox.go`, `outbox.go`, `handler.go`. The producer side is partially wired into `order`. Extend rather than replace.
- `app/catalog/cmd/server/main.go` for Dapr secret fetch retry loop — reuse the same `12 × 5s` pattern for sidecar publish readiness.

### Schema changes
- None for the broker itself.
- Confirm `outbox` and `inbox` tables exist per service (ent migration). If not, add via the existing schema package.

### Events
- **Emits:** infra-only — no domain events introduced here.
- **Consumes:** infra-only.

### API surface
- New internal HTTP route per service for Dapr subscription delivery. Not user-facing.

## Risks & mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Redis Streams ops complexity at scale | low (now) | med (future) | Document scale-out plan; Dapr lets us swap component without code change. |
| Dapr sidecar restarts mid-publish lose in-flight messages | med | low | Outbox owns durability; relay retries until acked. |
| CloudEvent envelope mismatch between producer and consumer | med | high | Single shared marshal/unmarshal in `pkg/eventbus`; integration test covers round-trip. |
| Subscription routes accidentally exposed to public ingress | low | high | Mount under an internal-only path prefix; ingress rule denies it. |
| Relay backlog grows unbounded during outage | med | med | Add a Prometheus alert on `outbox_relay_lag_seconds > 60`; auto-page. |

## Test plan

- **Unit:** envelope marshal/unmarshal round-trip; relay retry loop with simulated broker errors.
- **Integration:** testcontainers Redis + a minimal Dapr binary in CI, OR a custom in-process pubsub adapter behind the same `eventbus` interface for fast tests. End-to-end test: producer outbox row → relay → broker → consumer inbox → handler invoked exactly once.
- **E2E:** `tilt up`, observe `order.created` round-trip via the smoke consumer.
- **Manual smoke:** kill Redis pod mid-test; assert producer RPCs continue succeeding; restart Redis; assert backlog drains.

## Open questions

- Pin a fixed CloudEvents version (1.0) or follow whatever Dapr defaults to? **Decision:** pin 1.0 in our envelope marshaler so a Dapr upgrade can't silently break us.
- Topic naming convention: `service.entity.action` (e.g. `order.created`) — the `order` service already uses this; codify it in `pkg/eventbus`.
- Dead-letter queue strategy: drop, log, or republish to a `*.deadletter` topic? Default to `*.deadletter` + alert.

- Once shipped, every later phase that needs eventing should use `pkg/eventbus` — no service should call Dapr SDK directly from biz.
