# P0.3 — Stock reservation via order.created

## Problem

`catalog.product.stock_count` is informational. It's never decremented by orders and is never checked at order creation. Two concurrent orders for the last unit of a SKU both succeed, then a human notices when fulfillment fails. `order` already publishes `order.created` via the transactional outbox in `app/order/internal/biz/order.go`, but no consumer is wired. We have all the plumbing (`pkg/outbox`, ent, Dapr `pubsub` component) and an emitter; we lack the consumer + reservation domain.

## User stories

- As a **shopper**, I want my order to be rejected immediately if any line item is out of stock, so that I don't sit on a fake "confirmed" order that fails downstream.
- As a **fulfillment operator**, I want stock to decrement at order creation rather than at shipment, so that the live `stock_count` reflects what's actually available to sell.
- As a **platform engineer**, I want the reservation to be idempotent on `(order_id, product_id)`, so that retries from the outbox/inbox don't double-decrement.

## Acceptance criteria

- [ ] A new `stock_reservation` ent schema exists with at minimum `id`, `order_id`, `product_id`, `quantity`, `status` (`reserved`, `released`), `created_at`, with a unique index on `(order_id, product_id)`.
- [ ] `catalog` consumes `order.created` via `pkg/outbox.Inbox` (idempotent on `event_id`).
- [ ] On consume, `catalog` opens a single transaction that: a) inserts a `stock_reservation` row, b) decrements `product.stock_count` by line quantity, c) publishes either `stock.reserved` or `stock.rejected` via the outbox in the same transaction.
- [ ] If any line item has insufficient stock, the entire reservation is rejected — no partial reservations. The transaction rolls back and `stock.rejected` is published with a `reason` enum (`INSUFFICIENT_STOCK`, `PRODUCT_NOT_FOUND`, `PRODUCT_INACTIVE`).
- [ ] Re-delivery of the same `order.created` event (same `event_id`) is a no-op — verified by an inbox-dedup integration test.
- [ ] `stock.reserved` payload contains `order_id`, list of `{product_id, quantity}`, and `reserved_at`. `stock.rejected` adds `reason` and the offending `product_id`.
- [ ] Concurrent reservations for the last unit produce exactly one success and the rest fail with `INSUFFICIENT_STOCK` — verified by an integration test that fires N parallel orders.

## Out of scope

- Releasing reservations on order cancellation — handled in P0.4 (saga).
- Multi-warehouse stock or backorders — deferred to P1.6.
- Soft-reservation (e.g. cart-add reserve) — explicit no.
- Reservation expiry / TTL — explicit no for now; cancellation is the only release path.

## Implementation notes

### Files / packages
- `app/catalog/internal/data/ent/schema/stock_reservation.go` (new)
- `app/catalog/internal/biz/stock.go` (new) — `StockUsecase.Reserve(ctx, orderID, items)` returning `(reserved []Item, rejected *Reason, err error)`.
- `app/catalog/internal/data/stock.go` (new) — repo impl using ent + outbox in one transaction.
- `app/catalog/internal/server/event.go` (new) — Dapr pubsub HTTP handler that delegates to inbox + biz.
- `app/catalog/internal/biz/events.go` — add `TopicStockReserved`, `TopicStockRejected`, payload structs.
- Wire updates in `app/catalog/cmd/server/wire.go`.

### Pattern reference
- `app/order/internal/biz/order.go` shows transactional outbox pattern (`OutboxPublisher.Publish` inside the same `TxExecer`).
- `pkg/outbox/inbox.go` for dedup semantics.
- `docs/saga.md` for compensating-action design.

### Schema changes
- New table `stock_reservations` (ent). Migration is auto-applied on startup.
- New index `unique (order_id, product_id)`.
- No change to `products`; we mutate `stock_count` in place.

### Events
- **Emits:** `stock.reserved`, `stock.rejected` (both via outbox, in the same tx as the stock mutation).
- **Consumes:** `order.created` from `app/order/internal/biz/events.go`.

### API surface
- No external RPCs. Internal: Dapr HTTP subscription endpoint for the `pubsub` component.

## Risks & mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Hot-row contention on a popular SKU | high | med | Use `SELECT … FOR UPDATE` per row; benchmark with N=50 concurrent orders; add row-level retry with bounded backoff. |
| Outbox + ent in one tx leaks the broker error | med | high | `pkg/outbox` already enrolls the publish into the caller's `TxExecer`; verify with integration test that broker outage doesn't break reservation. |
| Lost `order.created` if Dapr broker drops | med | high | Outbox relay retries indefinitely with backoff; inbox dedupes on retry. Covered by P0.5. |
| Negative `stock_count` if migration of existing data is wrong | low | high | Add a CHECK constraint via ent annotation: `stock_count >= 0`. Test migration on a snapshot. |

## Test plan

- **Unit:** `StockUsecase.Reserve` table-driven — happy path, insufficient stock, missing product, partial line failure rolls back all.
- **Integration:** testcontainers Postgres + Redis Streams (or whatever P0.5 picks). Fire N=50 parallel `order.created` events for the last unit; assert exactly one `stock.reserved` and N-1 `stock.rejected`. Replay the same event ID twice; assert exactly one mutation.
- **E2E:** Place an order via the web app for a 1-stock product, confirm UI shows confirmed; place a second; confirm UI shows rejected.
- **Manual smoke:** `make dev`, seed catalog, place two concurrent orders via grpcurl, observe events on the broker.

## Open questions

- Does P0.5 land on Redis Streams, NATS, or Kafka? Affects the inbox dedup mechanism (Redis stream IDs vs. Kafka partition+offset vs. message ID).
- Should `stock_reservation.status` carry `released` from day one even though P0.4 handles release later? Yes — keeps the migration story simple.
- Reserve at `order.created` or wait until `order.confirmed` (after payment)? **Decision:** at `order.created`. Rationale: prevents oversell during the payment window; cost is a higher cancellation rate. Revisit if business sees abandoned-payment-spam pattern.
