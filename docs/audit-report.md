# Project Audit Report — 2026-05-01

## Executive Summary

The go-mall monorepo is a well-structured four-service Kratos microservices project with clean layering (`service → biz → data`), proper ent ORM usage, consistent Dapr secret injection, and a reasonable test suite. The most critical concerns are: (1) all four services share one logical Postgres database with no per-service isolation enforced by anything except separate DSN keys in the secret store; (2) no idempotency keys on any state-changing endpoint — duplicate order/payment/cart mutations will create duplicate rows; (3) the Dapr pub/sub component is declared but never wired, leaving the event-driven path entirely absent; (4) the `catalog` service carries an unprotected admin seed/clean endpoint (`/v1/admin/seed` and `/v1/admin/clean`) with no auth guard in code; (5) the payment service injects its JWKS URL via `os.Setenv` rather than wiring it through configuration, creating a race condition risk at startup. Ent, Go language patterns, and ecommerce architecture are otherwise solid; the biggest operational risks are data duplication under network retries and the absent outbox/saga infrastructure.

### 2. Saga / Distributed Transaction Issues

| # | Severity | Service | File | Issue | Recommendation |
|---|----------|---------|------|-------|----------------|
| S1 | Critical | order, payment | `internal/data/order.go`, `internal/data/payment.go` | No Saga or distributed transaction pattern is implemented. The checkout flow is: (1) create order, (2) create payment, (3) caller manually calls `UpdateOrderStatus` with `PAID`. If step 2 or 3 fails after step 1, the order is permanently stuck in PENDING with no compensating action. There is no rollback, no event-driven choreography, no orchestration. | Implement a checkout saga: order-created event → payment service creates payment → payment-completed event → order service marks paid. Define compensating actions (cancel order on payment failure). |
| S2 | Critical | all | `cmd/server/main.go` (all 4) | No outbox pattern. There is zero pub/sub publishing in any service. The Dapr pubsub component is declared but never used. State changes (order created, payment completed) are written directly to the DB with no event emission. Any downstream consumer of these events has no source of truth. | Implement the transactional outbox pattern: write events to an `outbox` table in the same ent transaction as the business write, then publish via Dapr `PublishEvent`. |
| S3 | Critical | all | all service layers | No idempotency keys anywhere. `CreateOrder`, `CreatePayment`, `AddItem`, `ClearCart` — all state-changing RPCs are not idempotent. A network timeout causing a client retry will create duplicate orders or payments. | Add an `idempotency_keys` table per service (or shared key store). Accept an `Idempotency-Key` header, check + upsert in the same DB transaction as the business write. |
| S4 | High | order | `internal/biz/order.go:92-110` | `UpdateStatus` has no guard against invalid state transitions — it accepts any valid enum value and will happily transition a DELIVERED order back to PENDING via a raw API call. Only `Cancel` enforces a transition guard. | Implement a state machine in `biz/order.go` that validates the full transition matrix before delegating to the repo. |
| S5 | Medium | all | none | No Dapr Workflow (`durabletask-go`) worker is registered anywhere. The reference pattern for multi-step flows requires a workflow engine for durability and retry. | For future saga implementation, register a Dapr Workflow worker in `main.go` and define workflow + activity functions. |
| S6 | Medium | all | none | No retry policies with jitter on any external call. The Dapr sidecar retry loop in `main.go` uses a fixed 5-second sleep with no backoff or jitter. | Use `cenkalti/backoff/v4` (already in `go.sum` transitively) for exponential backoff with jitter on the Dapr sidecar connect loop. |

### 3. Go Language Pitfalls

| # | Severity | Service | File | Issue | Recommendation |
|---|----------|---------|------|-------|----------------|
| G1 | Low | all | `go.mod:3` | Go version is `1.26.0`. Loop variable capture (the classic `for i, v := range` capture bug) was fixed in Go 1.22. All loop constructs observed in non-generated code correctly use `for i := range o.Items` index indexing, so there is no actual loop capture bug here. The Go version mitigates the classic pre-1.22 capture problem. | No action needed; the Go version covers this. |
| G2 | Medium | catalog | `internal/data/category.go:129-130` | `isUniqueViolation(err)` calls `err.Error()` on a potentially nil error. If a nil error were passed (defensive programming gap), this would panic. The call sites always check `err != nil` before calling this function, so no actual crash — but the function itself is unsafe in isolation. | Add a nil guard: `if err == nil { return false }`. |
| G3 | Low | payment | `cmd/server/main.go:112` | `os.Setenv("KEYCLOAK_JWKS_URL", v)` is called in `main()` before `wireApp()`. While this is single-threaded at this point, it sets a process-global environment variable. If any concurrent startup goroutine reads this env var between the `Setenv` and the `NewHTTPServer`/`NewGRPCServer` call, there is a TOCTOU window. More fundamentally, this pattern circumvents the clean config injection used by all other services; `catalog` passes `JwksUrl` through the `conf.Server.Auth` struct directly. | Pass the JWKS URL through `bc.Server.Auth.JwksUrl` (same pattern as catalog) instead of `os.Setenv`. Remove the env var dependency in `payment/internal/server/`. |
| G4 | Medium | all | all service `data.go` | No goroutine leak risk observed — no goroutines are spawned in business logic. The ent generated code manages its own connection pool. Context is consistently passed as the first parameter throughout. | No action needed. |
| G5 | Medium | order, payment | `internal/data/order.go:86-100`, `internal/data/payment.go:100-115` | Both `UpdateStatus` implementations do a query-then-update (two round-trips) without a transaction. Between the `Query` and the `Update`, another request could change the record. For the payment refund, this creates a TOCTOU race: two concurrent refund requests could both read `status=COMPLETED` and both successfully set `status=REFUNDED`. | Wrap the fetch + update in a single ent transaction (`client.Tx`) or use an atomic `UpdateOneID(...).Where(status == "COMPLETED").SetStatus("REFUNDED")` conditional update. |
| G6 | Medium | cart | `internal/data/cart.go:55-130` | `AddItem` performs multiple separate DB calls (query cart, query item, create/update item, re-query cart) without a transaction. Under concurrent requests for the same session, partial state is observable. The code does handle the `ConstraintError` race case manually (lines 110-124), but this is a multi-statement optimistic pattern that could still lose updates or return stale state. | Wrap the entire `AddItem` operation in a single ent transaction using `client.WithTx`. |
| G7 | Low | all | `internal/service/*.go` | Error wrapping: biz/service errors use kratos `errors.BadRequest(...)` / `errors.NotFound(...)` directly, not `fmt.Errorf("%w", ...)`. This is idiomatic for kratos but means `errors.Is` / `errors.As` on the returned errors will not traverse the chain in standard library fashion. The reference doc expects `%w` wrapping; here the kratos errors package is used instead, which provides its own `.Is()` implementation. | This is acceptable within the kratos ecosystem, but callers outside kratos middleware will not be able to use `errors.Is(err, biz.ErrOrderNotFound)` reliably. Add `%w` wrapping in the data layer when propagating ent errors to biz errors. |
| G8 | Low | all | `internal/server/http.go`, `internal/server/grpc.go` | Both `NewHTTPServer` and `NewGRPCServer` call `keyfunc.NewDefault(...)` independently in catalog, and each creates a separate JWKS key fetcher. For payment, the JWKS fetch happens via env var. This means two keyfunc instances, each with their own HTTP client and cache, per service. | Share a single `keyfunc.KeyfuncHandle` instance across both servers, constructed once and injected via Wire. |

### 4. E-commerce Microservices Pitfalls

| # | Severity | Service | File | Issue | Recommendation |
|---|----------|---------|------|-------|----------------|
| M1 | High | all | `deploy/k8s/base/secrets/secret.yaml:7` | All four services use the same `DATABASE_CONNECTION_STRING` secret key as their shared fallback, pointing to the same `dbname=mall` database. No per-service database isolation is enforced at the DB level — only the service-specific DSN key (`CART_DATABASE_CONNECTION_STRING` etc.) provides isolation, and even those point to the same host. If service-specific keys are absent (as in the k8s `secret.yaml` — which only defines the shared key), all services share one database schema. | Provision per-service databases (`mall_catalog`, `mall_cart`, `mall_order`, `mall_payment`) and create per-service k8s secrets. Each service's secret should ONLY have that service's key. |
| M2 | Critical | order, payment | all service/biz files | No idempotency on state-changing endpoints. `CreateOrder` and `CreatePayment` will create duplicate records on client retry. `AddItem` has a partial constraint guard at the DB level but no API-level idempotency key. | Add `Idempotency-Key` header support to all POST/PUT endpoints that create or mutate state. Persist `(key, response)` in the same DB transaction. |
| M3 | High | all | all service `server/http.go` | No rate limiting or backpressure middleware. The kratos HTTP/gRPC servers have no rate limiter configured. A burst of requests will hit the DB directly. | Add `github.com/go-kratos/aegis/ratelimit` (already in `go.sum`) as middleware in `NewHTTPServer` for each service. |
| M4 | High | catalog | `internal/server/http.go:55-56` | `POST /v1/admin/seed` and `DELETE /v1/admin/clean` are registered with no authentication guard in application code. The comment says "restrict at ingress," but there is no ingress rule enforcing this in the provided k8s manifests. A misconfigured ingress exposes data-wipe capability to anonymous callers. | Add at minimum a shared secret or IP allowlist check in `HandleSeed`/`HandleClean`, or gate them behind the same JWT middleware as write endpoints. Do not rely solely on ingress policy for security-sensitive operations. |
| M5 | Medium | all | `cmd/server/main.go` (all 4) | No distributed tracing exporter configured. `tracing.TraceID()` and `tracing.SpanID()` are injected into log fields, but no OpenTelemetry exporter (Jaeger, Zipkin, OTLP) is configured. Traces are not exported anywhere. The Dapr runtime can propagate W3C trace context, but the application layer is not emitting spans to a collector. | Configure an OTEL exporter (e.g., `go.opentelemetry.io/otel/exporters/otlp/otlptrace/otlptracegrpc`) in each `main.go` and set the global tracer provider. The Dapr sidecar annotations support `dapr.io/config` pointing to a tracing configuration. |
| M6 | Medium | all | k8s YAML | No circuit breaker on any inter-service call. Since services currently make no cross-service calls, this is not actively dangerous, but the absence of a circuit breaker pattern means any future cross-service call will have no fault isolation. | Use Dapr's resiliency policies (`resiliency.yaml`) to define circuit breakers and retry policies declaratively, or use `go-kratos/aegis/circuitbreaker` as middleware. |
| M7 | Medium | order | `internal/biz/order.go:92-97` | `UpdateStatus` accepts any valid status string without enforcing the transition rules. A caller can send `PENDING` → `DELIVERED` directly, bypassing `PAID` and `SHIPPED`. Only the `Cancel` path enforces a guard. | Encode a full transition table in `isValidTransition(from, to string) bool` and check it before delegating to the repo. |
| M8 | Low | catalog | `deploy/k8s/base/app/catalog.yaml` | The `catalog` deployment YAML has no `resources` block (CPU/memory requests and limits). The `cart`, `order`, `payment` deployments all have resource limits. Missing resource limits for `catalog` means it can OOM-kill adjacent pods or starve other workloads. | Add the same `resources` block present in `cart.yaml` to `catalog.yaml`. |
| M9 | Low | all | all k8s YAML | No `securityContext` on the `catalog` container. `cart`, `order`, `payment` all have `runAsNonRoot: true`, `readOnlyRootFilesystem: true`, `allowPrivilegeEscalation: false`. The `catalog` deployment lacks these security hardening settings. | Add the same `securityContext` block from `cart.yaml` to `catalog.yaml`. |
| M10 | Low | all | all k8s YAML | No `ConfigMap` for `catalog`. `cart`, `order`, `payment` each have an inline `ConfigMap` in their YAML. `catalog` has a `volumeMounts` referencing `catalog-config` but no corresponding `ConfigMap` in the provided manifest. This would cause the pod to fail to start with a missing ConfigMap. | Add the `catalog-config` ConfigMap to `catalog.yaml` (same pattern as `cart.yaml`). |

### 5. Dapr Integration Issues

| # | Severity | Service | File | Issue | Recommendation |
|---|----------|---------|------|-------|----------------|
| D1 | High | all | `deploy/k8s/base/app/*.yaml` | None of the k8s Deployment YAMLs include `dapr.io/app-port` annotation. Dapr uses this to know which port the application listens on for pub/sub subscription callbacks and service invocation. Without it, Dapr cannot route incoming events or service invocations to the app. | Add `dapr.io/app-port: "8000"` and `dapr.io/app-protocol: "http"` (or `"grpc"` for gRPC-native) to each deployment's annotations. |
| D2 | High | all | none | Pub/sub is completely unwired. The `pubsub` Dapr component (`pubsub.redis`) is declared in `deploy/k8s/base/infra/dapr/pubsub.yaml` but no service calls `dapr.PublishEvent()` or registers `AddTopicEventHandler()`. As stated in CLAUDE.md, events are absent by design, but this creates a significant gap for any eventual event-driven workflow. | When adding event publishing, use `daprClient.PublishEvent(ctx, "pubsub", "topic-name", payload)` from within the data layer after the transactional write. |
| D3 | Medium | catalog | `internal/data/dapr.go` | `NewDaprClient()` creates a raw gRPC connection without mTLS. In production, Dapr sidecar-to-app communication should use mTLS managed by Dapr, but the app-side gRPC connection here uses `insecure.NewCredentials()`. This client is currently only used for future pub/sub (the secret retrieval uses the main client created in `main.go`). | Remove `NewDaprClient` from the Wire graph until pub/sub is actually wired. When re-added, use `dapr.NewClient()` which uses the Dapr-managed port and credentials rather than constructing a raw gRPC connection manually. |
| D4 | Medium | all | `cmd/server/main.go` (all 4) | The Dapr retry loop uses a fixed 5-second sleep — not exponential backoff with jitter. If all 4 service pods restart simultaneously (e.g., after a node eviction), they will all hit the Dapr sidecar simultaneously at exactly 5-second intervals, creating thundering herd conditions. | Use jittered exponential backoff: e.g., `sleep = min(5s * 2^attempt, 60s) + rand(0, 1s)`. |
| D5 | Medium | payment | `cmd/server/main.go:112` | JWKS URL is passed via `os.Setenv` rather than through the conf struct. All other services that use JWKS (`catalog`) pass it through `bc.Server.Auth.JwksUrl`. The payment service sets an env var and then reads it in `server/http.go` and `server/grpc.go`. This is inconsistent and creates a process-global side effect. | Remove `os.Setenv` from `main.go`. Add `Auth *Server_Auth` to payment's `conf.proto` and pass the JWKS URL through the config struct the same way catalog does. |
| D6 | Low | all | `deploy/k8s/base/infra/dapr/` | No Dapr access control policy (`spec.accessControl`) is configured. Any service can invoke any other service via Dapr. | Add a `Configuration` Dapr resource with `spec.accessControl.defaultAction: deny` and explicit `allowedOperations` per service. |
| D7 | Low | all | `deploy/k8s/base/infra/dapr/` | No Dapr tracing configuration (`spec.tracing`) is defined. While the Dapr sidecar will auto-propagate W3C trace context, no sampling rate or exporter endpoint is configured for the sidecar-level traces. | Create a `dapr.io/v1alpha1/Configuration` resource with `spec.tracing` pointing to your OTLP endpoint and set `samplingRate: "1"` for full trace capture in dev. |
| D8 | Low | all | `deploy/k8s/base/secrets/secret.yaml:7` | The k8s Secret uses `stringData` (plaintext in YAML). While Kubernetes Secrets are base64 encoded at rest, the source YAML file contains the raw connection string committed to the repo. In production, this file should be gitignored and the Secret created by sealed-secrets, Vault, or External Secrets Operator. | Move the secret values out of the repo. Use Sealed Secrets or ESO. The checked-in file should have placeholder values only. |

### Critical Issues (data integrity / production crash risk)
- **S1**: No Saga — order/payment multi-step flow has no compensating actions
- **S2**: No outbox pattern — state changes emit no events despite pubsub component existing
- **S3**: No idempotency keys — duplicate records on retry (also tracked as M2)
- **M2**: Duplicate-create risk on all state-changing POST endpoints

### High Issues (correctness under real load)
- **E1**: Auto-migrate only — destructive schema changes will fail silently in production
- **S4**: Order `UpdateStatus` has no transition guard — any status can be written directly
- **M1**: Shared database secret — no enforced per-service DB isolation at the DB level
- **M3**: No rate limiting — DB exposed to unbounded request bursts
- **M4**: Unauthenticated seed/clean endpoints — data wipe callable anonymously if ingress misconfigured
- **G5**: Query-then-update TOCTOU race on payment refund and order status update
- **G6**: Multi-statement AddItem without a transaction — concurrent cart operations can corrupt state
- **D1**: Missing `dapr.io/app-port` annotation — Dapr cannot route callbacks to the app
- **D2**: Pub/sub completely absent

### Quick Wins (low effort, high value)
1. Add `dapr.io/app-port: "8000"` to all four k8s Deployment annotations (D1)
2. Add `resources` and `securityContext` to `catalog.yaml` to match other services (M8, M9)
3. Add the missing `catalog-config` ConfigMap to `catalog.yaml` (M10)
4. Fix payment JWKS URL to use conf struct instead of `os.Setenv` (D5, G3)
5. Wrap `AddItem` and `UpdateStatus` in ent transactions (G5, G6)
6. Replace `isUniqueViolation` string matching with proper pq error code check (E2)
